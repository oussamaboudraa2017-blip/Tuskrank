   388→## 11. Row Level Security (RLS) Audit
   389→
   390→❌ **RLS is NOT enabled** — there is no `ALTER TABLE … ENABLE ROW LEVEL SECURITY` or `CREATE POLICY` in any SQL file.
   391→
   392→| Check                                | Status |
   393→| ------------------------------------ | ------ |
   394→| `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` | ❌ None present |
   395→| `CREATE POLICY ... ON ...`           | ❌ None present |
   396→| `security_invoker = true` on views   | ✅ 9/9 plain views |
   397→| `SECURITY DEFINER` audit writer      | ✅ `fn_write_audit` |
   398→| `SET search_path = public` on plpgsql functions | ✅ All 13 plpgsql functions |
   399→
   400→**Implication:** The database is **RLS-ready** but not RLS-enforced. Any Supabase PostgREST client with table-level grants can currently read/write all rows. The schema shape and immutability (generated columns, RESTRICT FKs) provide defence in depth, but **Row Level Security MUST be enabled in a follow-up sprint** before exposing tables to authenticated client roles.
   401→
   402→This is **explicitly documented** in `README.md`, `DATABASE_REVIEW.md`, and `triggers.sql` as a Sprint-1.1 follow-up.
